# Phase 1.5 Results — Supervisor × RTO comparison (Wood-Berry, R1–R7)

Model: `qwen3:30b` (agentic), seed 42, 240-min scenarios, RTO + agent cadence 60 min, MPC 1 min.
Tables are generated by `analysis/build_tables.py` from `experiments/outputs/phase1_5/`; CSVs in
`analysis/tables/`. **Read the findings honestly** — the agent did not add measurable value in
this run (see §Findings); the most interesting results are a sensor-fault RTO failure (R6) and an
MA-GP adaptation limitation.

## Data completeness

All 8 configurations × 7 scenarios are present and parse cleanly **after one repair**:

- `qwen3_30b/baseline_ma-gp` crashed mid-**R3** on the pod (0-byte `R3.png`, no `log.json`; the
  per-config `set -e` killed R3–R7). Re-running it locally with the current code completed cleanly
  (R3 exit 0) — the pod failure was transient, not a code bug. Because baselines are **deterministic**
  (no LLM, seed 42, identical code), `baseline_ma-gp/R1–R7` was regenerated locally to complete the
  matrix. All other 7 configs are the pod's batch outputs (`03c1979`).

No empty/corrupt `log.json` remain.

## Findings (Step-4 questions, answered from the data)

### 1. Does the agent add value over the baseline?
**No — not in this run.** `agentic_*` and `baseline_*` final realized states are **bit-identical**
(dxD = 0.000000, dxB = 0.000000) in all 7 scenarios for both MA and MA-GP. The agent took a
consequential action **zero** times: 0 actions in all 7 MA scenarios; in MA-GP it issued one
`trigger_rto_run` on R1 and one on R7, both on the **final** cycle (t=240) and both with no effect
on the endpoint. With `qwen3:30b` the agent *observes* (it runs the read tools every cycle and
produces reasoned summaries) but does not act — the same "observe-but-don't-act" pattern seen
earlier with `qwen3:4b`, now confirmed at 30B scale. **The Phase-1 "agent adds measurable value"
claim is not supported by this data.**

### 2. Does the agent outperform the rule-based supervisors?
The agent and the rule-based supervisors reach the **same endpoints**, but for opposite reasons.
The rule-based supervisors *do* act — Rule 1 (|innovation| > 5e-4) fires on R1, R2, R6, R7 — but
their actions (extra `trigger_rto_run`s on an already-tracking RTO) **did not change the final
state** (e.g., `rule_naive_ma` R1 xD = 0.9610 = baseline). So:
- **R1 (slow drift):** rule-based acts 3× and adds 3 RTO solves; the agent holds. If holding is the
  correct response to a benign slow drift that MA absorbs, the agent shows better *restraint* — but
  this is the agent's universal passivity, not a demonstrated decision.
- **R6 (sensor fault):** rule-based fires once on the *phantom* innovation (over-reacting to a
  sensor lie); the agent holds. Again favourable to the agent *if* the hold were a diagnosis — but
  the agent holds everywhere, so we cannot credit it with diagnosing the sensor fault.
- **R3/R4/R5:** all supervisors agree (0 actions); no differentiation.

**Honest read:** neither the agent nor the rule books changed outcomes; the rule books add RTO
overhead and occasionally over-react, the agent does nothing. There is no scenario where the agent
*acted* and thereby beat a rule book.

### 3. MA vs MA-GP
This is the **largest** effect in the dataset, and it is a methodological caveat rather than a GP
result. Classical **MA adapts** within the scenario: under R3 (steam spike) it drives xD to 0.921
and under R4 (demand cap) to 0.987 — tracking the shifted economic optimum. **MA-GP does not adapt**:
its final xD sits at 0.955–0.960 in *every* scenario, and `converged` is False everywhere. The cause
is structural — MA-GP's trust-region BO uses `n_initial_samples = 6` global LHS points before it
begins acquisition, but a 240-min scenario at a 60-min RTO cadence yields only ~5–6 RTO solves, so
MA-GP **never leaves its seeding phase**. Within these scenario lengths MA-GP effectively "holds near
nominal." The MA-vs-MA-GP differences therefore measure *MA adapting vs MA-GP not adapting*, **not**
the quality of the GP surrogate. To compare GP value fairly, MA-GP needs many more RTO cycles
(longer scenarios or a faster RTO cadence) so it exits seeding — flagged as a limitation.

### 4. Supervisor behaviour patterns
- **Consistently trigger actions (across MA & MA-GP):** none. Under MA the rule books act on
  R1/R2/R6/R7; under MA-GP they act on none (the MA-GP innovation signature stays below the Rule-1
  threshold). The agent acts almost never.
- **All supervisors hold:** R3, R4, R5 (the RTO/event-recompute or infeasibility leaves nothing for
  a supervisor to add).
- **Supervisors disagree:** R1, R2, R7 under MA — the rule books act, the agent holds. (The
  naive-vs-smart pair never disagrees: Rule 2 never fired.)

### 5. R5 infeasibility handling — confirmed for all 8 configurations
- RTO correctly returned `converged = False` at every R5 (Table 5). ✅
- The system held at the previous valid operating point — MA configs at xD ≈ 0.966 (the pre-R5
  setpoint), MA-GP at ≈ 0.960 (nominal). No NaNs, no crash (the earlier R5 fix). ✅
- No supervisor took an inappropriate action (0 actions on R5 for all four supervisors). ✅

### 6. R6 — a genuine RTO failure mode (the most interesting result)
Under MA, the R6 analyzer bias (+0.05 on the xD reading, t=100–140) drives the RTO from
`converged=True` (t=0, 60) to **`converged=False / "infeasible"` at t=120, and it stays infeasible
at t=180 and t=240 even after the sensor bias clears at t=140.** The biased measurement corrupts the
modifier-adaptation estimate, and the filtered modifiers persist, so the RTO does not recover within
the scenario. **No supervisor caught or corrected this:** the agent held; the rule books fired Rule 1
once (on the phantom innovation) but did not restore convergence. This is exactly the vulnerability a
sensor scenario is meant to expose, and currently *nothing in the stack handles it* — a strong
motivation for genuine supervisory diagnosis (and for the data-reconciliation layer planned later).

## Limitations to flag honestly (Step 6)

- **The agent does not act.** With both `qwen3:4b` (earlier diagnostics) and `qwen3:30b` (this batch),
  the LLM supervisor reads state and reasons but rarely issues a tool action, and the few it issued
  were inconsequential (final-cycle triggers). The Phase-1 value claim is unsupported as-is; getting
  the agent to act will likely require prompt/tool-design changes (e.g., baseline-relative signals,
  an explicit "act when X" directive, or earlier trigger cadence) — and those changes must be made
  *before* over-claiming. The two MA-GP actions also illustrate stochasticity: the agent acted on
  R1/R7 but not the economically-clearer R3.
- **Agent acting on misread magnitudes.** The R1 MA-GP final message claimed the "RTO returned
  infeasible" on a scenario that is not infeasible — consistent with the earlier observation that the
  small model mis-reads the tiny innovation/economic magnitudes. Treat agent rationales as unreliable.
- **MA-GP never leaves seeding** at this scenario length/cadence, so any MA-vs-MA-GP claim is
  premature; it reflects non-adaptation, not GP performance.
- **Supervisor distinctions are mostly below the noise floor.** Endpoints within an RTO variant differ
  by ≤ ~0.001 in xD across supervisors — at or near the 2e-4 sensor noise — so most supervisor
  comparisons are not statistically meaningful here. The meaningful effects are MA-vs-MA-GP (large)
  and the R6 convergence failure (qualitative).
- **One config (`baseline_ma-gp`) was regenerated locally** after a transient pod crash; it is a
  deterministic baseline (no LLM), so this is reproducible, but it was not produced in the same batch
  run as the rest.
